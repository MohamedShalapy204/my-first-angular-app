Templates and Data Binding Deep Dive

You learned that components have templates. Now let's master how Angular templates work — the syntax that makes your components dynamic and interactive.

Key Insight: Angular templates aren't just HTML with mustaches. They're a powerful DSL (Domain Specific Language) that extends HTML with data binding, structural directives, and template expressions.

The Three Binding Types

Angular offers three primary data binding mechanisms. Understanding when to use each is crucial:

Binding Type Syntax Direction Use Case
Interpolation {{ expression }} Component → Template Displaying values in HTML
Property Binding [property]="expression" Component → Template Setting DOM properties
Event Binding (event)="handler" Template → Component Responding to user actions

Interpolation: More Than You Think

<!-- Basic interpolation -->
<h1>{{ product.name }}</h1>

<!-- With expressions -->
<p>Price: {{ product.price | currency }}</p>
<p>{{ 'Hello, ' + user.name + '!' }}</p>

<!-- With method calls (use sparingly!) -->
<p>{{ getDiscountedPrice(product) }}</p>
Performance Warning: Angular runs change detection to update interpolations. Avoid expensive calculations in templates — compute them in the component class instead.
Think About It: Why might {{ calculateComplexValue() }} in a template cause performance issues? What's the alternative?

Property Binding Deep Dive

Property binding sets DOM element properties, not attributes. This distinction matters:

<!-- Setting the disabled property (boolean) -->
<button [disabled]="isSubmitting">Submit</button>

<!-- Setting the src property -->
<img [src]="product.imageUrl" [alt]="product.name">

<!-- Setting a custom component property -->
<app-user-card [user]="currentUser" [isAdmin]="true"></app-user-card>

<!-- Binding to attribute (when no DOM property exists) -->
<div [attr.role]="'button'">Click me</div>

Attribute vs Property

  • Property: DOM element property (JavaScript object property)
  • Attribute: HTML attribute (initial value, doesn't update)
<!-- This sets the HTML attribute, not the DOM property -->
<input value="Initial">

<!-- This sets the DOM property, which updates dynamically -->
<input [value]="currentValue">
Practice Exercise: Create a component with a text input. Use property binding to set its value from a signal. What happens when you type in the input? How would you make it two-way bound?

Event Binding: Handling User Actions

<!-- Basic click event -->
<button (click)="addToCart()">Add to Cart</button>

<!-- With event object -->
<button (click)="handleClick($event)">Click</button>

<!-- Other events -->
<input (input)="search($event)">
<input (keydown.enter)="submitSearch()">
<div (mouseleave)="hideTooltip()"></div>

<!-- Filtering events -->
<input (keyup.enter)="submit()">

Event Object Access

handleClick(event: MouseEvent) {
  console.log('Clicked at:', event.clientX, event.clientY);
  console.log('Target:', event.target);
}
$event: Angular provides the native DOM event object as $event in event bindings. For component events (Output), it's the value you emitted.

Two-Way Binding: The Sugar Syntax

Two-way binding combines property and event binding:

<!-- Two-way binding with ngModel -->
<input [(ngModel)]="username">

<!-- What it's actually doing: -->
<input [value]="username" (input)="username = $event.target.value">
Modern Angular: With signals, two-way binding is evolving. The [(ngModel)] still works, but signals provide a more reactive approach.

Template Expressions

Angular evaluates expressions in a limited context for security:

<!-- Allowed: -->
{{ product.name }}
{{ items.length }}
{{ isLoggedIn ? 'Welcome' : 'Please login' }}
{{ getFullName(user) }}

<!-- NOT allowed (for security): -->
{{ constructor() }}        <!-- No new operators -->
{{ window.location }}      <!-- No global objects -->
{{ something.click() }}    <!-- No assignment expressions -->

Expression Context

  • Can access: Component properties, template variables, imported modules
  • Cannot access: Global JavaScript objects, assignment statements, new operator

1. {{ Math.PI }}
2. {{ items.push('new') }}
3. {{ user.getName() }}
4. {{ value = 5 }}

Template Variables

Angular templates can have local variables:

<!-- Using *ngFor with template variable -->
<ul>
  <li *ngFor="let item of items; let i = index; let odd = odd">
    {{ i }}. {{ item }} {{ odd ? '(odd)' : '' }}
  </li>
</ul>

<!-- Using @for (modern syntax) -->
<ul>
  @for (item of items; track item.id; let i = $index; let odd = $odd) {
    <li>{{ i }}. {{ item.name }} {{ odd ? '(odd)' : '' }}</li>
  }
</ul>

Template Reference Variables

Get direct references to DOM elements or components:

<!-- Reference an input element -->
<input #emailInput type="email">
<button (click)="validateEmail(emailInput.value)">Validate</button>

<!-- Reference a component -->
<app-child #childComponent></app-child>
<button (click)="childComponent.doSomething()">Call Child</button>

<!-- Using with ViewChild decorator -->
@Component({...})
export class ParentComponent {
  @ViewChild('emailInput') emailInput!: ElementRef<HTMLInputElement>;
  
  focusEmail() {
    this.emailInput.nativeElement.focus();
  }
}
#variable: Creates a reference available only within the template. Use @ViewChild in the class to access it programmatically.

Common Pitfalls

1. Forgetting Change Detection

// ❌ This might not update
@Component({...})
export class CounterComponent {
  count = 0;
  
  increment() {
    this.count++; // Works with OnPush only if signal or manual detection
  }
}

// ✅ Better with signals
@Component({...})
export class CounterComponent {
  count = signal(0);
  
  increment() {
    this.count.update(n => n + 1); // Always triggers update
  }
}

2. Mutating Arrays/Objects

// ❌ This won't trigger update
this.items.push(newItem);

// ✅ Create new reference
this.items = [...this.items, newItem];
With Signals: Signals handle immutability differently. You can mutate and still get updates, but immutability is still recommended for predictability.
