Component Lifecycle & Signals

You know how to build components and bind data. Now let's understand when things happen in a component's life, and how Angular's modern signals make state management elegant.

Key Insight: Components have a lifecycle with specific hooks. Signals provide a reactive alternative to traditional state management, making updates automatic and efficient.

The Component Lifecycle

Every Angular component goes through a series of steps from creation to destruction:

Creation → Change Detection → Destruction ↓ ↓ ↓ Constructor → Lifecycle Hooks → ngOnDestroy ↓ ↓ @Input() ngOnChanges @ViewChild ngOnInit ngAfterViewInit etc.

Essential Lifecycle Hooks

Hook When It Runs Common Use Cases
constructor Class instantiation Dependency injection only
ngOnChanges After @Input changes React to parent changes
ngOnInit After first ngOnChanges Component initialization
ngAfterViewInit After view initializes Access @ViewChild elements
ngOnDestroy Before destruction Cleanup subscriptions

The Complete Lifecycle

@Component({...})
export class UserComponent implements OnInit, OnDestroy {
  private subscription!: Subscription;
  
  constructor(private userService: UserService) {
    // 1. Constructor - inject dependencies only
    console.log('Constructor');
  }
  
  ngOnChanges(changes: SimpleChanges) {
    // 2. On Changes - when @Input properties change
    console.log('Changes:', changes);
  }
  
  ngOnInit() {
    // 3. On Init - component initialized
    console.log('Component initialized');
    this.subscription = this.userService.getUsers()
      .subscribe(users => this.users = users);
  }
  
  ngAfterViewInit() {
    // 4. After View Init - view ready
    console.log('View ready');
    // Safe to access @ViewChild here
  }
  
  ngOnDestroy() {
    // 5. On Destroy - cleanup
    console.log('Cleaning up');
    this.subscription.unsubscribe();
  }
}
Common Mistake: Don't put initialization logic in the constructor. Use ngOnInit instead. The constructor runs before Angular sets up inputs and view children.
Think About It: Why does Angular separate constructor from ngOnInit? What would break if you accessed @ViewChild in the constructor?

Angular Signals: Modern Reactivity

Signals are Angular's answer to reactive state management. They automatically track dependencies and update the UI when state changes.

Basic Signals

import { Component, signal, computed } from '@angular/core';
    
@Component({...})
export class CounterComponent {
  // Signal with initial value
  count = signal(0);
  
  // Computed signal (derived state)
  doubleCount = computed(() => this.count() * 2);
  
  // Method to update
  increment() {
    this.count.update(n => n + 1);
    // Or: this.count.set(this.count() + 1);
  }
}

Why Signals Are Different

  • Automatic tracking: No manual change detection needed
  • Efficient updates: Only re-renders what depends on changed signals
  • Computed values: Automatically recompute when dependencies change
  • Easy testing: Simple to mock and test

Signal Patterns

// Writable signal
const user = signal<User>({ name: 'Alice', age: 30 });
user.set({ name: 'Bob', age: 25 });
user.update(u => ({ ...u, age: u.age + 1 }));

// Computed signal (read-only, derived)
const displayName = computed(() => `${user().name} (${user().age})`);

// Effect (side effects)
effect(() => {
  console.log('User changed:', user());
  // Automatically re-runs when user() changes
});

// Signal from observable
const users = toSignal(this.userService.getUsers(), { initialValue: [] });
Signal vs Observable: Signals are synchronous and push-based. Observables are asynchronous and pull-based. Use signals for UI state, observables for async operations.

Signals in Templates

@Component({
  template: `
    <div>
      <h2>Count: {{ count() }}</h2>
      <p>Double: {{ doubleCount() }}</p>
      <button (click)="increment()">+1</button>
      
      <!-- Automatic updates -->
      @if (count() > 5) {
        <p>Count is greater than 5!</p>
      }
      
      <!-- List with signals -->
      @for (item of items(); track item.id) {
        <div>{{ item.name }}</div>
      }
    </div>
  `
})
export class SignalDemoComponent {
  count = signal(0);
  doubleCount = computed(() => this.count() * 2);
  items = signal<Item[]>([]);
  
  increment() {
    this.count.update(n => n + 1);
  }
}

When to Use Each Pattern

Scenario Use Why
Simple component state Signals Automatic updates, minimal boilerplate
Async data streams Observables Handle HTTP, WebSocket, intervals
Complex state logic Signals + computed Derived state without manual tracking
Form state Reactive Forms Validation, async validators

Lifecycle with Signals

@Component({...})
export class UserComponent implements OnInit {
  user = signal<User | null>(null);
  
  // Effect runs when user signal changes
  constructor() {
    effect(() => {
      const currentUser = this.user();
      if (currentUser) {
        console.log('User loaded:', currentUser.name);
      }
    });
  }
  
  ngOnInit() {
    // Load user after component initializes
    this.userService.getCurrentUser()
      .subscribe(user => this.user.set(user));
  }
}

Common Patterns

1. Cleanup Pattern

export class DataComponent implements OnInit, OnDestroy {
  private destroy$ = new Subject<void>();
  
  ngOnInit() {
    this.dataService.getData()
      .pipe(takeUntil(this.destroy$))
      .subscribe(data => this.data = data);
  }
  
  ngOnDestroy() {
    this.destroy$.next();
    this.destroy$.complete();
  }
}

2. Signal with Initial Data

export class ListComponent {
  items = signal<Item[]>([]);
  loading = signal(true);
  
  constructor() {
    // Load data immediately
    this.loadData();
  }
  
  async loadData() {
    this.loading.set(true);
    const items = await this.api.getItems();
    this.items.set(items);
    this.loading.set(false);
  }
}
